# ADR-032: Tripartite consensus for safety-critical edge decisions

## Status
PROPOSED

## Context
On NEXUS vessels, certain decisions are safety-critical and cannot be made by a single subsystem without risk of cascading failure. For example, the decision to deploy new bytecode that controls actuators involves the safety validator (is the bytecode safe?), the trust engine (do we trust the source?), and the mission runner (is this the right time?). If any one of these subsystems has a transient error or corrupted state, a single-point-of-failure decision could lead to unsafe actuator commands. We need a decision-making pattern that ensures no single faulty component can authorize a dangerous action, while remaining efficient enough to run on resource-constrained Jetson hardware.

## Decision
Adopt a **tripartite consensus** model for safety-critical decisions. Three independent evaluation channels must all agree before a high-impact action is authorized: (1) **Safety Validator** — verifies bytecode structural integrity, opcode validity, stack safety, and cycle budget compliance; (2) **Trust Engine** — checks that the requesting subsystem's trust score exceeds the minimum threshold for the requested action level; (3) **Mission Authority** — confirms the deployment is on the current mission queue and matches the expected source. For non-critical decisions (telemetry commits, trust event recording), single-channel authorization suffices. The consensus check is implemented as a simple AND gate: all three must return `True` within a 5-second timeout. If any channel vetoes, the action is denied and an incident is logged. This pattern is applied to bytecode deployment, autonomy level changes, and actuator mode transitions.

## Consequences
**Positive:** Eliminates single-point-of-failure for dangerous actions; clear separation of concerns between safety, trust, and mission logic; easy to audit which channel denied an action; adds defense-in-depth without significant compute overhead. **Negative:** Increases latency for authorized actions (must wait for all three channels); any channel timeout defaults to deny, which may cause false negatives during high system load; adds architectural complexity. **Mitigation:** Timeout is generous (5s) to accommodate busy subsystems; non-critical actions bypass consensus; channel health is monitored and timeout events trigger YELLOW incidents.
