# ADR-031: Hybrid trust persistence (git commits for history + in-memory for speed)

## Status
PROPOSED

## Context
The NEXUS trust engine needs both a persistent, auditable trust history and sub-millisecond trust score lookups during reflex execution on the edge. Writing every trust event to disk or git on every heartbeat cycle would introduce unacceptable latency—trust scores are consulted multiple times per reflex evaluation. However, a purely in-memory trust model loses all history on reboot, making it impossible to reconstruct why a vessel's autonomy was reduced or to audit trust-based decisions after an incident. The maritime environment demands both: real-time reflex safety checks and post-incident forensic review.

## Decision
Implement a **hybrid trust persistence** model with two tiers. The **hot tier** is an in-memory trust engine (the existing INCREMENTS-based engine in `jetson/trust/`) that maintains current scores per subsystem and responds to lookups in O(1). The **cold tier** is git commits in `.agent/trust/` that record every trust event as a structured JSON file with metadata (subsystem, event type, severity, timestamp, computed score). The hot tier processes events immediately for real-time reflex decisions; the cold tier is updated asynchronously during the REMEMBER phase of the heartbeat cycle. On system boot, the cold tier is replayed (most recent N events per subsystem) to reconstruct hot-tier state. This provides both auditability and performance, with the git history serving as an immutable, append-only trust ledger that can be reviewed by the fleet orchestrator or human operators.

## Consequences
**Positive:** Sub-millisecond trust lookups during reflex execution; full trust history preserved in git for auditing; natural integration with the git-agent bridge; trust state survives reboot via git replay. **Negative:** Eventual consistency between hot and cold tiers (events may be lost if the system crashes between heartbeat cycles); replay on boot adds startup latency proportional to event history; memory usage grows with subsystem count. **Mitigation:** Flush critical trust events (RED-level) immediately to cold tier; cap replay window to most recent 100 events per subsystem; use memory-mapped structures for hot tier if subsystem count grows beyond ~50.
