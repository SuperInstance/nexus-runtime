# ADR-035: Bytecode safety validation as PR pipeline (6-stage)

## Status
PROPOSED

## Context
Deploying bytecode to NEXUS vessels that control physical actuators on maritime vessels is inherently dangerous. A single unsafe instruction—an out-of-bounds memory access, an infinite loop, an unclamped actuator command—could cause physical damage or loss of vessel control. The current validation in BytecodeDeployer checks structural properties but does not enforce a complete safety pipeline before deployment reaches the vessel. As NEXUS scales to more vessels and more AI-generated reflexes, we need a rigorous, multi-stage validation pipeline that gates deployment the way a PR review gates code merge, ensuring no unsafe bytecode can reach production vessels.

## Decision
Implement **bytecode safety validation as a 6-stage PR pipeline** that must pass all stages before a bytecode deployment is authorized. The stages are: (1) **Structural Validation** — checks bytecode alignment (8-byte instructions), valid opcodes (0x00-0x56), and non-empty payload; (2) **Stack Safety Analysis** — simulates stack operations to verify no overflow (max 16) or underflow occurs; (3) **Control Flow Verification** — ensures all jump targets are within bounds and detects unreachable infinite loops via cycle budget check; (4) **Data Integrity Check** — rejects NaN and Infinity in float operations; (5) **Actuator Safety Gate** — requires CLAMP_F instruction before any WRITE_PIN, ensuring all actuator outputs are bounded; (6) **Trust Policy Check** — verifies the requesting agent's trust score meets the minimum for the requested action level and that the bytecode hash matches what was reviewed (if from a PR). Each stage produces a pass/fail result with specific error messages. The pipeline fails fast—stages are evaluated in order and stop on first failure. This pipeline runs both locally (on the Jetson agent) before deployment and in the cloud (via the git-agent PR review) for fleet-dispatched bytecode, providing defense-in-depth.

## Consequences
**Positive:** Comprehensive safety coverage from structure to semantics to trust; fails fast with clear error messages; dual enforcement (edge + cloud) prevents bypass; each stage is independently testable; pipeline can be extended with additional stages. **Negative:** Adds latency to deployment (milliseconds for structural checks, seconds for full pipeline); stage 6 (trust policy) depends on up-to-date trust scores; strict pipeline may reject valid but unconventional bytecode patterns. **Mitigation:** Stage 5 (CLAMP_F requirement) can be relaxed for read-only bytecode; trust score caching in the hot tier makes stage 6 fast; pipeline results are cached per bytecode hash to avoid redundant validation.
