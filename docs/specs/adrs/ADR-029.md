# ADR-029: Git-native modular architecture (monorepo core + polyrepo ecosystem)

## Status
PROPOSED

## Context
NEXUS operates across multiple vessels, each running independent firmware and edge software, yet requiring fleet-wide coordination. A pure monorepo would force every vessel to carry unrelated code; a pure polyrepo would make shared core evolution painful and create versioning chaos. We need a structure where the core NEXUS runtime (VM, wire protocol, safety state machine) evolves in a single monorepo for atomic cross-layer commits, while each vessel's deployed artifacts—bytecode, telemetry logs, trust history—live in their own git repositories as the source of truth for that vessel's operational state. The fleet orchestration layer (git-agent) needs to consume both the monorepo's releases and each vessel's operational repo without fragile cross-repo linking.

## Decision
Adopt a **monorepo core + polyrepo ecosystem** model. The `/tmp/nexus-runtime` repository serves as the monorepo containing the firmware C code, the Jetson Python control layer, shared schemas, and documentation. Each vessel gets its own dedicated git repository (created per-vessel by the fleet orchestrator) that stores `.agent/` directories with bytecode deployments, telemetry commits, trust event logs, and safety incident records. The git-agent acts as the bridge, cloning the monorepo's release artifacts into vessel repos and pushing operational state back. Version alignment is maintained by embedding the monorepo commit hash in each deployment manifest.

## Consequences
**Positive:** Atomic evolution of core runtime layers; clear separation of concerns between "what NEXUS is" (monorepo) and "what each vessel is doing" (polyrepo); natural audit trail per vessel; vessels can operate fully offline with their own repo. **Negative:** Requires discipline to keep monorepo and vessel repos in sync; tooling must handle two git paradigms; onboarding new vessels requires repo provisioning. **Mitigation:** The NexusBridge abstracts repo management; deployment manifests carry version provenance.
