# ADR-036: Emergency protocol bridge (NEXUS safety → git-agent fleet alert)

## Status
PROPOSED

## Context
NEXUS vessels can encounter emergency situations—sensor failures, trust collapse, E-Stop events, communication loss—that require immediate coordinated response across the fleet. Currently, safety events are logged locally and reported to the bridge, but there is no structured escalation path that connects the vessel's emergency state to fleet-wide coordination. When a vessel enters a RED emergency state, the fleet needs to: (1) know immediately (not wait for the next telemetry sync), (2) understand the severity and category, (3) take coordinated action (dispatch assessment, activate backup vessel), and (4) track the incident lifecycle from detection through resolution. Without a dedicated emergency bridge, critical events are mixed with routine telemetry and may be missed or delayed.

## Decision
Implement the **Emergency Protocol Bridge** as a dedicated subsystem within the NEXUS agent that monitors vessel state through five specialized detectors (SensorFailure, TrustCollapse, CommunicationLoss, SafetyViolation, MissionTimeout) and maintains a four-level emergency state machine (GREEN → YELLOW → ORANGE → RED). Each level triggers predefined automated responses: YELLOW increases monitoring and logs; ORANGE alerts the captain, reduces autonomy, and prepares contingencies; RED halts all autonomous operations, safes actuators, creates a GitHub Issue with RED label, commits the incident to git, reduces trust scores, notifies the fleet via structured alerts, and starts watchdog monitoring. The bridge produces a structured fleet alert (JSON) containing incident ID, severity, category, vessel state snapshot, trust snapshot, and recommended fleet actions. De-escalation is supported through incident resolution tracking—the emergency level automatically adjusts as incidents are resolved. The entire incident lifecycle is recorded in git for post-incident review.

## Consequences
**Positive:** Rapid, structured emergency response that doesn't depend on human intervention; clear escalation path from detection to fleet coordination; full incident lifecycle tracking in git; integrates naturally with the git-agent bridge for GitHub Issue creation and fleet notification; configurable thresholds allow tuning per vessel. **Negative:** Adds complexity to the agent; false positives (especially at YELLOW level) could cause alert fatigue; RED response halts all autonomous operations, which may be undesirable in certain scenarios; dependency on git bridge for some response actions. **Mitigation:** Thresholds are configurable per vessel and per detector; YELLOW incidents are logged but don't halt operations; RED response can be overridden by captain via direct console access; detector outputs include confidence indicators to reduce false positives.
