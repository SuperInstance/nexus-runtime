# ADR-033: CRDT-based offline fleet synchronization

## Status
PROPOSED

## Context
NEXUS vessels frequently operate in disconnected environments where direct network communication between vessels and the fleet coordinator is unavailable for hours or days. During these periods, multiple vessels may independently generate telemetry, trust events, and mission results that need to be merged without conflicts when connectivity is restored. Traditional synchronization approaches (last-write-wins, vector clocks with conflict resolution) either lose data or require a central authority to resolve conflicts. We need a synchronization mechanism that allows concurrent writes from multiple vessels and the fleet coordinator, guarantees eventual consistency without data loss, and operates within the git-native architecture where the unit of synchronization is a git repository.

## Decision
Implement **CRDT-based offline fleet synchronization** using operation-based conflict-free replicated data types mapped to the git commit structure. Each vessel's `.agent/` directory contains CRDT registers (for scalar values like trust scores) and CRDT OR-sets (for additive collections like telemetry readings and incident records). When offline, vessels commit to their local git repo as usual. When connectivity is restored, the git-agent performs a sync cycle: (1) fetch remote changes, (2) merge CRDT states using the CRDT merge function (commutative, associative, idempotent), (3) commit the merged state, (4) push. The CRDT merge is deterministic—regardless of merge order, all replicas converge to the same state. Trust scores use LWW-Register (last-writer-wins with vector clock timestamps) since only the latest score matters. Telemetry and incident logs use OR-Set (observed-remove set) to preserve all entries without duplication. The implementation leverages Python dicts with vector clock metadata, avoiding external CRDT library dependencies.

## Consequences
**Positive:** Zero data loss during extended offline periods; deterministic convergence regardless of sync order; natural fit with git's merge model; vessels can sync peer-to-peer or through the fleet coordinator. **Negative:** CRDT metadata adds storage overhead per record; merge conflicts at the git level still require resolution (mitigated by appending rather than modifying files); complex merge scenarios (concurrent trust score updates from both vessel and fleet) need careful timestamp comparison. **Mitigation:** Design the `.agent/` directory structure so files are always appended, never mutated; use high-resolution monotonic clocks for LWW-Register timestamps; limit CRDT complexity to well-understood data types.
