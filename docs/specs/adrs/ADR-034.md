# ADR-034: Skill cartridge format (JSON + bytecode + metadata)

## Status
PROPOSED

## Context
NEXUS vessels need a portable, self-describing package format for deploying autonomous capabilities—reflex behaviors, mission scripts, and sensor processing pipelines—to the edge. Currently, bytecode is deployed as raw binary with separate metadata files, requiring the deployer to reconstruct context from multiple sources. Fleet operators and AI agents need to create, share, and deploy capabilities without understanding the internal bytecode format. A standardized cartridge format would enable: (1) AI-generated skills to be packaged and validated before deployment, (2) a skill marketplace where proven capabilities can be shared between vessels, (3) versioned skill dependencies where one reflex can reference another, and (4) fleet-wide skill distribution via the git-agent bridge.

## Decision
Define the **Skill Cartridge Format** as a directory-based package with a required `manifest.json` at the root. A cartridge contains: (1) `manifest.json` — metadata including skill name, version, author (human or AI agent), description, input/output schemas, trust requirements (minimum trust scores per subsystem), safety constraints (max actuator range, cycle budget), and dependencies on other cartridges; (2) `bytecode.bin` — compiled NEXUS VM bytecode that has passed the 6-stage safety validation pipeline; (3) `bytecode.meta.json` — deployment metadata from the safety validator (hash, instruction count, stack depth, validation timestamp); (4) `README.md` — human-readable description and usage instructions. The cartridge is packaged as a gzipped tarball with the naming convention `{skill-name}-v{version}.cart.tar.gz`. Deployment via NexusBridge validates the manifest, checks trust requirements against the target vessel's current trust scores, and commits the extracted cartridge to `.agent/skills/{skill-name}/`. The format is intentionally simple (JSON + binary) to avoid introducing a build toolchain dependency.

## Consequences
**Positive:** Self-describing deployment packages; enables skill sharing and reuse; clear trust和安全 requirements declared upfront; compatible with git-based versioning and distribution. **Negative:** Additional packaging step in the deployment pipeline; manifest schema must be versioned to handle evolution; large skill collections increase repo size. **Mitigation:** Keep manifest schema minimal and backward-compatible; use git LFS or blob storage for large cartridges; the NexusBridge validates manifests before accepting any deployment.
